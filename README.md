# 网页版扫雷游戏

## 项目简介

网页版扫雷游戏是一个基于原生 HTML、CSS、JavaScript 实现的前端课程实践项目。项目不依赖后端服务，不使用构建工具，也不引入外部 CDN，下载后可以离线打开，并且适合直接部署到 GitHub Pages。

项目实现了经典扫雷的核心玩法：随机生成地雷、计算周围地雷数量、点击翻开格子、插旗标记、空白区域自动展开、胜负判断和最佳成绩记录。页面同时针对课程展示进行了视觉优化，包含顶部信息栏、状态提示、难度选择和移动端适配。

## 功能列表

- 支持三种难度：
  - 简单：9x9，10 个地雷
  - 中等：16x16，40 个地雷
  - 困难：16x30，99 个地雷
- 顶部信息栏显示当前难度、剩余地雷数、用时、最佳成绩和游戏状态。
- 点击“新游戏”可以重置棋盘、计时器、旗帜数量和游戏状态。
- 左键点击格子可以翻开。
- 电脑端右键点击格子可以插旗或取消插旗。
- 手机端长按格子可以插旗或取消插旗。
- 已翻开的格子不能再插旗，已插旗的格子不能被左键翻开。
- 点击地雷后游戏失败，并显示所有地雷。
- 翻开所有非地雷格子后游戏胜利，不要求旗子全部插对。
- 空白格子会自动展开周围安全区域。
- 数字 1-8 使用不同颜色，便于识别周围地雷数量。
- 未翻开、已翻开、旗帜、地雷状态有明显视觉区别。
- 使用 `localStorage` 按难度保存最短通关时间。
- 小屏幕下棋盘保持可点击大小，困难难度可横向滚动查看。

## 技术栈

- HTML5：负责页面结构和语义化内容。
- CSS3：负责页面布局、棋盘样式、响应式适配和状态视觉效果。
- JavaScript：负责游戏逻辑、事件处理、棋盘渲染、计时器和本地存储。
- localStorage：用于保存不同难度下的最佳成绩。
- GitHub Pages：用于静态网页部署。

## 本地运行方法

项目是纯静态网页，不需要安装依赖。

方法一：直接打开

1. 下载或克隆项目到本地。
2. 双击根目录下的 `index.html`。
3. 使用浏览器打开后即可开始游戏。

方法二：使用本地静态服务

在项目根目录执行：

```bash
python -m http.server 8000
```

然后在浏览器访问：

```text
http://localhost:8000
```

## GitHub Pages 部署方法

1. 将项目文件提交到 GitHub 仓库，确保 `index.html` 位于仓库根目录。
2. 打开 GitHub 仓库页面。
3. 进入 `Settings`。
4. 在左侧菜单中找到 `Pages`。
5. 在 `Build and deployment` 中选择 `Deploy from a branch`。
6. 分支选择 `main` 或 `master`，目录选择 `/root`。
7. 点击保存，等待 GitHub Pages 自动部署。
8. 部署完成后，通过 GitHub Pages 提供的网址访问项目。

本项目资源路径使用相对路径：

```html
<link rel="stylesheet" href="./assets/style.css">
<script src="./assets/app.js"></script>
```

因此适合在 GitHub Pages 中从仓库根目录直接访问。

## 核心算法说明

### 随机布雷

游戏开始时，根据当前难度确定行数、列数和地雷数量。程序通过 `Math.random()` 随机生成地雷坐标，并使用 `Set` 保存坐标字符串，例如 `row-col`。

使用 `Set` 的原因是可以避免同一个格子被重复放置地雷。当 `Set` 中的坐标数量达到当前难度要求的地雷数量后，再遍历这些坐标，将对应格子的 `isMine` 标记为 `true`。

### 计算周围地雷数

布雷完成后，程序会遍历整个棋盘。对于每一个非地雷格子，检查它周围 8 个方向的相邻格子：

- 左上、上、右上
- 左、右
- 左下、下、右下

每发现一个相邻地雷，就将计数加 1。最终结果保存在格子的 `adjacentMines` 属性中，用于翻开格子时显示数字。

### 空白区域自动展开

当玩家翻开一个周围地雷数为 0 的空白格时，程序会继续翻开它周围的格子。如果周围仍然有空白格，则继续向外展开。

本项目使用递归方式实现自动展开。为了避免重复访问和死循环，每个格子在进入递归展开前都会先标记为已翻开。后续如果再次遇到这个格子，会因为 `isRevealed` 为 `true` 而直接返回。

### 胜负判断

失败条件：玩家翻开的格子是地雷，游戏立即失败，并显示所有地雷。

胜利条件：已翻开的格子数量等于非地雷格子的总数量。也就是说，只要所有安全格子都被翻开，就判定游戏胜利，不要求玩家必须把所有地雷都正确插旗。

非地雷格子数量计算方式：

```text
安全格子数 = 行数 * 列数 - 地雷数
```

每成功翻开一个非地雷格子，已翻开数量加 1。当已翻开数量等于安全格子数时，游戏胜利。

## 时间复杂度和空间复杂度分析

设棋盘行数为 `R`，列数为 `C`，地雷数量为 `M`。

- 初始化棋盘：需要创建 `R * C` 个格子，时间复杂度为 `O(R * C)`，空间复杂度为 `O(R * C)`。
- 随机布雷：平均情况下需要生成 `M` 个不重复坐标，时间复杂度约为 `O(M)`，额外空间复杂度为 `O(M)`。
- 计算周围地雷数：需要遍历所有格子，每个格子最多检查 8 个方向，时间复杂度为 `O(R * C)`。
- 翻开单个数字格：只处理当前格子，时间复杂度为 `O(1)`。
- 空白区域自动展开：最坏情况下可能访问整个棋盘，时间复杂度为 `O(R * C)`。
- 胜负判断：通过已翻开数量和安全格子总数比较完成，时间复杂度为 `O(1)`。
- 整体空间复杂度：主要来自棋盘二维数组，空间复杂度为 `O(R * C)`。

## 常见问题

### 1. GitHub Pages 打不开页面怎么办？

先确认 GitHub Pages 是否已经部署完成。首次开启 Pages 后通常需要等待一段时间。如果仍然打不开，检查仓库 `Settings` -> `Pages` 中的分支和目录是否选择正确。

### 2. 打开 GitHub Pages 后显示 404 怎么办？

请确认 `index.html` 是否放在仓库根目录。如果 `index.html` 被放进了子文件夹，而 Pages 目录选择的是 `/root`，GitHub Pages 就无法从根目录找到入口文件。

### 3. 页面打开了，但样式或脚本没有加载怎么办？

检查 `assets/style.css` 和 `assets/app.js` 是否存在，并确认路径没有写错。本项目使用的是：

```html
<link rel="stylesheet" href="./assets/style.css">
<script src="./assets/app.js"></script>
```

如果文件夹名称或文件名被修改，需要同步修改 `index.html` 中的路径。

### 4. 为什么困难模式棋盘会出现横向滚动？

困难模式是 16x30，共 30 列。为了保证手机端格子不会被压缩到难以点击，项目保留固定格子大小，并允许棋盘区域横向滚动。

### 5. 为什么刷新页面后最佳成绩还在？

最佳成绩保存在浏览器的 `localStorage` 中。只要没有清除浏览器站点数据，记录就会保留。如果换浏览器或清除缓存，最佳成绩可能会消失。

### 6. 本地双击打开和使用本地服务有什么区别？

本项目没有后端接口，直接双击 `index.html` 就可以运行。使用本地静态服务更接近网页部署环境，适合在提交或部署前进行最终检查。
